# clean_main.py

`clean_main.py` это рыночный управляющий скрипт для ИЭС, который на каждом тике:

- читает текущее состояние энергосистемы через `ips.init()`;
- определяет текущий профицит или дефицит энергии;
- анализирует, как исполнились продажи на бирже на прошлом тике;
- выбирает базовую цену продажи;
- при профиците сначала продает энергию, а только потом отправляет остаток в накопители;
- при дефиците разряжает накопители;
- сохраняет состояние и пишет подробный лог тика.

Ниже описана именно текущая логика файла [clean_main.py](/Users/danya/store_data/clean_main.py).

## Главный принцип

Для принятия решений скрипт использует прямой сигнал API `psm.total_power.external`.

Смысл поля:

- `external < 0` означает, что система имеет профицит и отдает энергию наружу;
- `external > 0` означает, что система имеет дефицит и добирает энергию снаружи.

Поэтому канонические формулы в скрипте такие:

```python
balance_after_consumption = -psm.total_power.external
surplus_now = max(0.0, -psm.total_power.external)
deficit_now = max(0.0, psm.total_power.external)
```

Это главный источник истины для всей логики продажи, заряда и разряда.

## Формула точной выработки

Если нужно восстановить выработку из баланса, используется простая формула:

```python
generated_now = consumed_now + losses_now - external_now
```

Где:

- `consumed_now = psm.total_power.consumed`
- `losses_now = psm.total_power.losses`
- `external_now = psm.total_power.external`

Скрипт сохраняет в лог и сырое `GEN` из API, и диагностическое `GEN_FROM_BAL`, вычисленное по формуле выше. Это нужно только для контроля корректности данных. Для самой торговой логики используется именно `external`.

## Источники данных

Скрипт использует следующие поля стенда:

- `psm.total_power.generated`
- `psm.total_power.consumed`
- `psm.total_power.losses`
- `psm.total_power.external`
- `psm.exchange`
- `psm.exchangeLog`
- `psm.objects`

Из объектов используются в первую очередь накопители:

- фильтруются объекты типа `storage`;
- сломанные накопители пропускаются;
- для каждого накопителя читается текущий заряд `obj.charge.now`.

## Что делает скрипт на каждом тике

### 1. Загружает прошлое состояние

Скрипт читает файл `clean_state.json`.

Из него используются:

- `prev_tick`
- `prev_useful_supplied`
- `prev_storage_action`
- `prev_base_sell_price`
- `fill_ewma`
- `market_history`

Дополнительно `clean_main.py` смотрит в `hand_state.json`, если тот содержит более свежий тик. Это нужно, чтобы антидемпинг и cooldown не ломались при переключении между ручным и автоматическим скриптом.

### 2. Собирает снимок мощности

Функция `current_power_snapshot(psm)` строит компактный снимок:

- `generated`
- `generated_from_balance`
- `consumed`
- `losses`
- `external`
- `balance_from_external`
- `balance_after_consumption`
- `surplus_now`
- `deficit_now`

Важное правило:

- `surplus_now` и `deficit_now` считаются напрямую от `external`;
- они не выводятся повторно через `generated - consumed - losses`, хотя это эквивалентная проверочная формула.

### 3. Анализирует исполнение продаж

Скрипт проходит по `psm.exchange` и рассматривает только собственные заявки на продажу:

- `askedAmount`
- `contractedAmount`
- `instantAmount`
- `askedPrice`
- `contractedPrice`

Из этого считаются:

- общий заявленный объем продажи `sell_asked`
- общий контрактный объем `sell_contracted`
- объем мгновенного рынка `sell_instant`
- fill-rate `sell_fill_rate`
- средняя заявленная цена
- средняя реально исполненная цена

Также поддерживается короткая история рынка `market_history`, которая хранится в state.

### 4. Строит рыночные сигналы

На основе текущего тика и короткой истории скрипт определяет:

- `good_fill`
- `weak_fill`
- `near_zero_fill`
- долю ухода в мгновенный рынок
- серию плохих тиков
- быстрый и мягкий режим плохого рынка

Также обновляется `fill_ewma`, чтобы сгладить резкие скачки исполнения.

### 5. Выбирает опорную цену рынка

Скрипт строит `market_ref` из двух источников:

- средняя цена реально исполненных сделок на последних тиках;
- медиана последних значений `psm.exchangeLog`.

Если есть реальное исполнение, оно имеет больший вес. Если исполнения не было, используется лог биржи. Если и там пусто, используется `DEFAULT_SELL_PRICE`.

### 6. Выбирает базовую цену продажи

Базовая цена `base_sell_price` строится так:

- если на прошлом тике сделки реально исполнялись, берется `sell_avg_contracted_price - PRICE_UNDERCUT`;
- иначе берется `market_ref - PRICE_UNDERCUT`.

Далее цена корректируется:

- вверх, если рынок хорошо берет энергию;
- вниз, если fill слабый;
- еще агрессивнее вниз, если рынок почти не берет энергию;
- дополнительно ограничивается скоростью падения от прошлого тика через `MAX_BASE_PRICE_DROP_PER_TICK`.

### 7. Считает антидемпинговый лимит объема

Скрипт не пытается продать весь профицит без ограничений.

Лимит рассчитывается по прошлому полезному профициту:

```python
anti_dump_limit = prev_useful_supplied * ANTI_DUMP_FACTOR + ANTI_DUMP_ADDON
```

Если прошлый тик не соседний, используется стартовое значение `FIRST_TICK_ANTI_DUMP_LIMIT`.

`prev_useful_supplied` это прошлый положительный профицит системы, а не фактический проданный объем.

## Логика накопителей и продажи

### Если сейчас дефицит

Если `deficit_now > 0`, скрипт:

- не продает энергию;
- пытается закрыть дефицит разрядом накопителей;
- сначала старается не опускаться ниже `BASE_RESERVE_PER_STORAGE`;
- если этого не хватает, разрешает углубленный разряд;
- если на прошлом тике был заряд и cooldown еще активен, разряд пропускается.

### Если сейчас профицит

Если `surplus_now > 0`, скрипт действует строго в таком порядке:

1. Сначала продает:

```python
sell_amount_total = min(surplus_now, anti_dump_limit)
```

2. Потом отправляет остаток в накопители:

```python
unsold_useful = max(0.0, surplus_now - sell_amount_total)
```

Именно это поведение сейчас является основным: продажа приоритетнее заряда.

## Лестница заявок

Если объем продажи достаточно велик (`MIN_ORDER_ENERGY`), скрипт может не ставить одну заявку, а разбивать объем на лестницу.

Параметры лестницы:

- максимум 3 заявки;
- веса по умолчанию `[0.55, 0.30, 0.15]`;
- каждая следующая ступень ставится выше по цене;
- при плохом рынке ступени сжимаются и опускаются вниз.

Это позволяет:

- часть объема продать ближе к рыночной цене;
- часть попробовать отдать дороже;
- быстрее адаптироваться, если рынок перестал брать объем.

## Что реально отправляется на стенд

На выходе скрипт отправляет:

- `psm.orders.discharge(...)` для накопителей на разряд;
- `psm.orders.charge(...)` для накопителей на заряд;
- `psm.orders.sell(...)` для биржевых заявок на продажу.

Все объемы округляются вниз через `order_amount()`, чтобы не нарушать ограничения API.

## Что сохраняется в clean_state.json

После тика скрипт сохраняет:

- `prev_tick`
- `prev_useful_supplied`
- `prev_base_sell_price`
- `prev_storage_action`
- `fill_ewma`
- `market_history`

Назначение полей:

- `prev_useful_supplied` влияет на антидемпинг следующего тика;
- `prev_base_sell_price` ограничивает слишком резкие скачки цены;
- `prev_storage_action` нужен для cooldown после заряда;
- `fill_ewma` и `market_history` помогают адаптировать цену и лестницу к рынку.

- `EXTERNAL`
- `LOSSES`
- `BAL_FROM_EXT`

- `USEFUL_ENERGY_NOW`

Это основной файл для разбора поведения стратегии после игры.

## Роль TARGET_CHARGE

Сейчас `target_charge` в `clean_main.py` фактически равен:

```python
storage_count * BASE_RESERVE_PER_STORAGE
```

Он остается в коде и логах как информационный ориентир по резерву, но в текущей версии не управляет активным предзарядом. При профиците порядок жесткий: сначала продажа, затем заряд остатка.

## Текущие ограничения

Важные ограничения текущей реализации:

- скрипт не использует прогнозы потребления и генерации;
- приоритет полностью у текущего состояния и недавней рыночной истории;
- `target_charge` сейчас не ведет к отдельной стратегии предзаряда;
- если у системы хронический дефицит, скрипт только пытается разряжать накопители и не занимается более сложной перестройкой поведения.

## Короткий итог

Текущая версия `clean_main.py` это рыночный sell-first контроллер.

Его базовая логика:

- профицит и дефицит определяются напрямую через `psm.total_power.external`;
- при профиците сначала идет продажа, потом заряд остатка;
- при дефиците идет разряд накопителей;
- цена продажи адаптируется по недавнему исполнению заявок;
- антидемпинг ограничивает объем продажи от тика к тику.
